# CAD图形结构数据提取专项课题

## 📅 记录时间：2026-04-20（调整版）

---

## 一、课题背景与目标

### 背景
兴土股份在桥梁大临钢结构业务中，需处理大量设计院交付的CAD图纸（PDF/DWG格式）。目前痛点：
- 人工录入效率低、易出错
- 图纸分散，难以检索和综合分析
- 无法与可视化平台、BIM系统打通

### 目标
研发能从桥梁CAD图纸中自动提取图形结构数据的技术方案，实现：
1. 设计院交付图纸的**自动解析入库**
2. 既有图纸的**数字化存档检索**
3. 多项目**综合分析**（跨项目构件统计、算量等）

## 五、2026-04-20 关键发现

### 发现1：图纸数据提取完全可行

**验证结果**：
- DXF图纸6,356个实体全部可提取
- 579个块引用（标准构件）可统计
- 23个图层的实体分布清晰
- 文字标注、尺寸标注均可提取

**意义**：
> 数据提取方向正确，技术上完全可行

### 发现2：画施工图是错误方向

**问题**：
- EzDXF无法复制专业块定义（308实体的钢板桩块）
- 10种专业标注样式无法复现
- 施工图需要AutoCAD原版+测量数据

**结论**：
> ❌ 放弃"画施工图"这个错误方向  
> ✅ 聚焦"数据提取"这个正确方向

### 发现3：构件语义识别是核心缺失

**现状**：
- 我能提取几何数据（圆、直线、多段线）
- 但无法自动判断"这是桩基"还是"墩身"

**需要的规则**：

| 构件 | 识别特征 |
|------|---------|
| 桩基 | 图层"桩基"/"钢护筒"，圆形，较小半径 |
| 墩身 | 图层"墩身"，多段线封闭，较大尺寸 |
| 梁段 | 图层"梁段"，特定线型，跨度特征 |
| 盖梁 | 横向长条，多在墩顶 |
| 支座 | 圆形/矩形，较小，在梁和墩之间 |

## 七、构件识别引擎（待开发）

### 7.1 识别流程

```
几何数据
    ↓
提取特征（图层/形状/尺寸/位置）
    ↓
匹配规则（if-then规则库）
    ↓
构件分类（桩基/墩身/梁段/盖梁/支座）
    ↓
关联关系（父子/兄弟）
    ↓
结构化输出
```

### 7.2 识别规则示例

```python
# 桩基识别规则
def identify_pile(entity, layer_info):
    if layer_name in ['桩基', '钢护筒', 'pile']:
        if shape == 'CIRCLE' and radius < 2000:
            return 'pile_foundations'
    
# 墩身识别规则  
def identify_pier(entity, layer_info):
    if layer_name in ['墩身', 'pier']:
        if shape == 'LWPOLYLINE' and is_closed and area > 10000:
            return 'pier_body'

# 梁段识别规则
def identify_girder(entity, layer_info):
    if layer_name in ['梁段', '箱梁', 'T梁', 'girder']:
        if shape == 'LWPOLYLINE' and aspect_ratio > 3:
            return 'girder_segment'
```

### 7.3 规则配置表

| 构件 | 图层关键词 | 形状特征 | 尺寸范围 |
|------|-----------|---------|---------|
| 桩基 | 桩基,钢护筒,pile | 圆形 | R=500~2000mm |
| 墩身 | 墩身,pier | 封闭多段线 | 面积>1m² |
| 梁段 | 梁段,箱梁,T梁,girder | 多段线 | 长度>5m |
| 盖梁 | 盖梁,cap | 横向多段线 | 宽度>2m |
| 支座 | 支座,bearing | 圆形/矩形 | R<500mm |

## 八、数据模型（待实现）

### 8.1 构件表结构

```sql
CREATE TABLE components (
    id SERIAL PRIMARY KEY,
    project_id TEXT,          -- 项目ID
    drawing_id TEXT,          -- 图纸ID
    component_type TEXT,      -- 构件类型：桩基/墩身/梁段/盖梁
    component_name TEXT,       -- 构件名称：如"3#墩桩基"
    layer_name TEXT,           -- 原始图层
    geometry_data JSONB,       -- 几何数据
    dimensions JSONB,          -- 尺寸参数
    material_spec TEXT,        -- 材料规格
    quantity INTEGER,          -- 数量
    unit TEXT,                 -- 单位
    coordinate_range JSONB,    -- 坐标范围
    parent_id INTEGER,         -- 父构件ID
    metadata JSONB             -- 其他元数据
);
```

### 8.2 工程量统计视图

```sql
-- 按构件类型统计
CREATE VIEW quantity_summary AS
SELECT 
    component_type,
    COUNT(*) as total_count,
    SUM(quantity) as total_quantity,
    JSONB_OBJECT_AGG(DISTINCT material_spec, COUNT(*)) as material_breakdown
FROM components
GROUP BY component_type;
```

## 十二、结论

### 调整后的方向

1. **聚焦数据提取**：不追求画图，追求提取
2. **建立识别引擎**：规则引擎优先于AI模型
3. **强化数据结构化**：PostgreSQL + 空间数据
4. **简化输出形式**：示意图够用，施工图交给专业软件

### 核心认知

> **数据提取** ≠ **画图**
> 
> 我们的目标是：从CAD图纸中提取数据，不是用代码替代AutoCAD画图
