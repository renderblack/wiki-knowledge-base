# 3DGS 模型压缩方案

> 更新：2026-04-21
> 目标：压缩 524 国道施工场景 L20 高精度模型，支撑上云 + 网页浏览

---

## 一、数据分析结论

### 1.1 模型结构

| 项目 | 数据 |
|------|------|
| 项目 | 新塍至王江泾 524 国道 |
| Tile 总数 | 139 个 |
| 网格范围 | X: 1~20, Y: 1~18 |
| 目标 LOD | L20（最高精度，近景细节） |
| L20 文件数 | 137 个 |
| **L20 总体积** | **3.78 GB** |
| 单 Tile 点数 | 约 26~74 万高斯 |
| 每点属性 | 17 个（xyz, nxnynz, f_dc0~2, opacity, scale0~2, rot0~3） |

### 1.2 属性分析

| 属性 | 存储值范围 | 存储空间 | 关键发现 |
|------|-----------|---------|---------|
| x, y, z | 世界坐标 (world coords) | 12 bytes | ⚠️ 世界坐标值大（~562000），float16误差±12.5cm |
| nx, ny, nz | 全 0 | 12 bytes | ✅ **全部为 0，可删除** |
| f_dc_0~2 | [-1.77, +1.77] | 12 bytes | ✅ float16误差<0.0005，无损 |
| opacity | logit空间 [-12.9, +7.2] | 4 bytes | ✅ float16误差<0.004，无损 |
| scale_0~2 | ln空间 [-12, +1] | 12 bytes | ✅ float16误差<0.004，无损 |
| rot_0~3 | 四元数（模长=1） | 16 bytes | ✅ float16误差<0.0003，无损 |

### 1.3 坐标系关键发现

```
metadata SRSOrigin = 562378.8, 3407695.2, -5.0  (世界坐标，EPSG:4549)
metadata Box Center = 246.0, 398.3, 16.9         (tile局部坐标)
PLY x range = [191, 300] → 相对 SRSOrigin 的局部坐标
```

**问题**：PLY 存储世界坐标（x ≈ 191 = 562378 + 191），float16 在大数值（~562000）附近精度仅 ±12.5cm。

**解法**：坐标转为相对 Box Center 存储（变为 ±55m 范围），float16 精度提升至 ±0.75cm。

## 二、压缩方案

### 2.1 方案 B-Final（主推）：自定义 .3dgs 二进制格式

**核心思路**：
- 坐标转为**相对 Box Center** 存储（消除大数值精度问题）
- 删除全 0 法向量
- float16 存储所有属性
- 自定义二进制 .3dgs 格式

**属性编码表**：

| 属性 | 存储方式 | 字节/点 |
|------|---------|---------|
| x, y, z | float16（相对 Box Center） | 6 bytes |
| f_dc_0~2 | float16 | 6 bytes |
| opacity | float16 | 2 bytes |
| scale_0~2 | float16 | 6 bytes |
| rot_0~3 | float16 | 8 bytes |
| **合计** | | **28 bytes/点** |

**精度保证**：
- 位置（±55m 范围）：float16 误差 ±0.75cm ✅
- 色彩：float16 误差 < 0.0005 ✅
- 不透明度/尺度：float16 误差 < 0.004 ✅
- 旋转：float16 误差 < 0.0003 ✅
- **总体：无损压缩（精度高于视觉感知阈值）**

**压缩效果**：

| 阶段 | 单 Tile | L20 全量 |
|------|---------|---------|
| 原始 | 27.6 MB | **3.78 GB** |
| 删除法向量 | 23.4 MB | 3.21 GB |
| float16 量化 | 11.4 MB | 1.56 GB |
| **最终** | **~11 MB** | **~1.5 GB** |

**压缩比：2.4×（最终体积减少 60%）**

### 2.2 方案 A（备选）：PLY float32 标准格式

**思路**：不做数据类型转换，只删除法向量，保持 PLY 格式 100% 兼容。

| 属性 | 处理 |
|------|------|
| 位置 xyz | 保持 float32 |
| 法向量 nx/ny/nz | 删除 |
| 其他 | 保持 float32 |

- 压缩比：68→56 bytes = **1.2×**
- 优点：GridMaster 原生支持
- 缺点：压缩率低

### 2.3 方案 C（备选）：PLY uint16 量化

**思路**：所有属性转为 uint16 缩放存储，保持 PLY 格式兼容。

- 压缩比：68→22 bytes = **3.1×**
- 缺点：uint16 线性量化对 logit/scale/rot 等宽范围属性误差过大（不可接受）
- **不推荐**

## 三、文件格式设计

### 3.1 .3dgs 格式规范

```
Header (JSON, UTF-8):
{
  "version": "1.0",
  "n_vertices": 264453,
  "properties": ["x","y","z","f_dc_0","f_dc_1","f_dc_2","opacity","scale_0","scale_1","scale_2","rot_0","rot_1","rot_2","rot_3"],
  "box_center": [246.001152, 398.297043, 16.916382],
  "bbox_min": [191.4, 342.1, 2.0],
  "bbox_max": [300.6, 454.5, 31.9],
  "srs": "EPSG:4549",
  "srs_origin": [562378.833, 3407695.236, -4.975]
}

Data Section (binary, float16, little-endian):
  [x0,y0,z0, f_dc0_0,f_dc0_1,f_dc0_2, op0, s0_0,s0_1,s0_2, r0_0,r0_1,r0_2,r0_3]  ← 28 bytes
  [x1,y1,z1, ...]  ← 重复 n_vertices 次
```

### 3.2 Tile manifest

每个 Tile 生成一个 .3dgs 文件 + 一个同名的 .json（存储 header 信息）

## 五、为什么现有论文方案不适用

> 2026-04-21 补充

以下方案在学术界表现优秀，但在"直接压缩已训练的 .ply 模型文件"这一具体场景下均无法直接应用：

| 方案 | 问题 |
|------|------|
| 特征平面压缩（Triplane） | 需要在训练阶段介入，修改 3DGS 训练 pipeline，压缩发生在训练期间。我们的模型已训练完毕，无法套用。 |
| VQ/PCA 量化 | 针对特征场的训练期压缩，作用于 Splatting 过程。压缩对象是隐式特征，不是显式 .ply 属性。 |
| 前馈压缩网络 | 一次前馈完成压缩，看似可行，但依赖配套训练框架（需要同时有训练好的编码器）。同样需要原始训练数据。 |
| Octree 空间分区 | 我们的 Tile 分块本质已是空间分区，重复做 Octree 无额外收益。|
| Mipmap/LOD 金字塔 | 现有模型已有 LOD 分级（L10/L15/L20），进一步做 mipmap 层级收益有限。|

**结论**：在已训练 .ply 文件的前提下，最有效的手段是数据类型压缩（float32→float16）+ 属性裁剪（删除全0法向量）。这就是方案 B-Final 的核心。

## 六、风险与备选

| 阶段 | 风险 | 概率 | 备选方案 |
|------|------|------|---------|
| Phase 1 | GridMaster 不支持 .3dgs | 中 | 改用方案 A（.ply float32） |
| Phase 2 | 部分 tile 数据异常 | 低 | 脚本跳过异常 tile，记录日志 |
| Phase 2 | 转换中途断电 | 低 | 加 checkpoint，每 10 个 tile 保存进度 |
| Phase 3 | 质量验收不达标 | 低 | 改用 float32（方案 A），压缩率降至 1.2× |
| 通用 | float16 精度不够 | 极低 | 位置改用 float32（其他 float16），压缩率 2× |
